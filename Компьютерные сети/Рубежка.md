
# Глава 13. Адресация в стеке протоколов TCP/IP

## 13.1. Задачи адресации и масштабируемость

TCP/IP решает три основные задачи:

1.  Согласованное использование адресов разного типа: отображение IP‑адресов на локальные, доменных имён на IP‑адреса.
2.  Обеспечение уникальности адресов в нужном масштабе (компьютер, подсеть, Интернет).
3.  Конфигурирование сетевых интерфейсов и приложений.

Ключевое слово — **масштабируемость**. Процедуры назначения, отображения и конфигурирования адресов одинаково хорошо работают и в маленькой, и в огромной сети.

## 13.2. Структура стека TCP/IP (четыре уровня)

<img width="431" height="266" alt="image" src="https://github.com/user-attachments/assets/18f772bc-f3db-4006-90fe-227494b594ef" />

*Как найти: «tcp ip stack 4 layers», «стек TCP/IP 4 уровня»; на схеме снизу вверх: Network Interface, Internet (IP), Transport (TCP/UDP), Application (HTTP, DNS, SMTP).*

Уровни стека:

1.  **Уровень сетевых интерфейсов**
    *   Отвечает за упаковку (инкапсуляцию) IP‑пакета в кадр конкретной подсети (Ethernet, PPP, Wi‑Fi).
    *   Преобразует IP‑адреса в адреса технологии подсети (например, через ARP).
    *   Любая среда, позволяющая IP‑узлам общаться без промежуточных маршрутизаторов, рассматривается IP как одна **линия связи (Link)**.

2.  **Сетевой уровень (Internet)**
    *   Главный протокол — **IP** (межсетевой протокол). Дейтаграммный, без соединений, «по возможности».
    *   Функционирует на всех хостах и маршрутизаторах.
    *   Вспомогательные протоколы: **ICMP** (сообщения об ошибках), **IGMP** (управление группами), протоколы маршрутизации (**RIP, OSPF**).

3.  **Транспортный уровень**
    *   **TCP**: гарантированная доставка с установлением логического соединения, нумерацией байтов, квитанциями и восстановлением порядка.
    *   **UDP**: доставка дейтаграмм без гарантий, без соединений.

4.  **Прикладной уровень**
    *   Соответствует трём верхним уровням OSI.
    *   Сервисы: FTP, telnet, SMTP, HTTP.

## 13.3. Типы адресов стека TCP/IP

Три типа:

1.  **Локальные (аппаратные) адреса**
    *   MAC‑адрес в Ethernet, номера виртуальных каналов в ATM. Действуют только в пределах подсети.

2.  **Сетевые адреса (IP‑адреса)**
    *   Глобальная система адресации, уникальная в пределах всей составной сети.
    *   **IPv4:** 4 байта (32 бита). Пример: `192.45.66.17`.
    *   Состоит из номера сети и номера узла.

3.  **Символьные (доменные) имена**
    *   Удобны для людей. Пример: `base2.sales.zil.ru`.
    *   Называются **DNS‑именами**.

Соответствие между типами адресов устанавливается только через таблицы, так как между ними нет функциональной зависимости.

## 13.4. Формат и классы IP‑адресов (IPv4)

Длина адреса — 4 байта, записывается как `w.x.y.z`, например `128.10.2.30`.

Нужно деление на номер сети и номер узла. Есть три исторических подхода, сейчас используется **бесклассовая адресация (CIDR)** с масками, но базовая идея идёт от классов.

*   **Класс E** — зарезервированы.

**Особые IP‑адреса**

Запрещено использовать номера сетей и узлов из одних нулей или единиц. Это даёт специальные адреса:

*   **0.0.0.0** (в адресе назначения не используется) — обозначает данный узел.
*   **Номер сети = 0** — узел назначения в той же сети.
*   **255.255.255.255** (все единицы) — **ограниченное широковещание** (только в своей подсети).
*   **Номер сети.все единицы** — широковещание в указанную сеть (directed broadcast).
*   **127.x.x.x** — адрес **обратной петли (loopback)** для внутреннего тестирования стека.

## 13.5. Использование масок при IP‑адресации

Маска — число, двоичная запись которого содержит непрерывную последовательность единиц в тех разрядах, которые относятся к **номеру сети**.

*   Маска **255.255.255.0** — для класса C (`/24`).
*   Маска **255.255.0.0** — для класса B (`/16`).
*   Маска **255.0.0.0** — для класса A (`/8`).

## 13.6. Порядок назначения IP‑адресов

*   **Централизованное распределение:** Орган ICANN → региональные регистраторы (ARIN, RIPE, APNIC) → крупные провайдеры → мелкие провайдеры → клиенты. Так получаются **глобальные/публичные адреса**.
*   **Частные адреса:** Рекомендованы для изолированных (автономных) сетей.
    *   Класс A: `10.0.0.0`
    *   Класс B: `172.16.0.0 – 172.31.0.0`
    *   Класс C: `192.168.0.0 – 192.168.255.0`

## 13.7. Технология бесклассовой маршрутизации (CIDR)

**CIDR** (Classless Inter‑Domain Routing) решает две проблемы: более экономное расходование адресов и сокращение таблиц маршрутизации.

*   Отказ от классов, маска может быть любой длины.
*   Адресный блок записывается как `адрес/префикс`, где префикс — число единиц в маске.
    *   `193.20.30.0/28`: 28 бит — номер сети, 4 бита — для 16 узлов.
*   **Агрегирование адресов:** Все адреса одного провайдера имеют общий префикс. В таблицах внешних маршрутизаторов вместо многих записей хранится одна строка на префикс.
*   **Супернетинг (supernetting):** обратная операция к subnetting, объединение нескольких подсетей в одну.

## 13.8. Отображение IP‑адресов на локальные адреса. Протокол ARP

Для доставки IP‑пакета через локальную сеть (напр., Ethernet) нужен MAC‑адрес следующего узла. Его находит **ARP (Address Resolution Protocol)**.

[IMG: 13.5 — Схема работы протокола ARP]
*Как найти: «arp request reply diagram», «схема работы ARP»; показан запрос broadcast (кто имеет IP X?), и ответ unicast (я имею IP X, мой MAC — Y).*
<img width="855" height="521" alt="image" src="https://github.com/user-attachments/assets/12eac703-1574-4705-9675-95b8e32349c9" />

1.  IP‑модуль даёт задание ARP: «Узнать MAC для IP1».
2.  ARP проверяет свою **ARP‑таблицу** (кэш).
3.  Если записи нет — отправляется **широковещательный ARP‑запрос** по локальной сети.
4.  Узел, узнавший свой IP, отвечает **ARP‑ответом** со своим MAC‑адресом.
5.  Полученное соответствие заносится в ARP‑таблицу (динамическая запись, имеет срок жизни).

## 13.9. Доменная служба имен DNS

**DNS (Domain Name System)** — это распределённая база данных, отображающая символьные имена в IP‑адреса (и наоборот).

Дерево имён начинается с **корня (.)**. Составные части отделяются точкой. Пример: `home.microsoft.com`. (com — домен верхнего уровня, microsoft — второго, home — имя хоста).

**Типы серверов DNS**

*   **Корневые серверы (Root):** хранят адреса серверов доменов верхнего уровня (TLD: .com, .ru, .org).
*   **Серверы TLD:** хранят адреса серверов, отвечающих за домены второго уровня (напр., `cisco.com`).
*   **Полномочные серверы (Authoritative):** хранят отображения имён для своей **зоны** в файлах зоны.

Файл зоны содержит записи типов:
*   `A` — отображение имени в IPv4.
*   `AAAA` — отображение имени в IPv6.
*   `NS` — имя DNS‑сервера для домена.
*   `MX` — имя почтового сервера.

**Процедуры разрешения имени**

[IMG: 13.9 — Процедуры разрешения имён DNS]
<img width="717" height="436" alt="image" src="https://github.com/user-attachments/assets/a609d2a1-b0c6-4c29-af74-99c2c903e9a0" />

*Как найти: «dns recursive iterative query», «рекурсивный итеративный dns запрос»; три схемы: итеративная (клиент сам опрашивает серверы), рекурсивная (сервер опрашивает за клиента), смешанная.*

*   **Итеративная:** Клиент сам опрашивает цепочку DNS‑серверов, пока не получит ответ.
*   **Рекурсивная:** Клиент отправляет запрос локальному DNS‑серверу, который берёт всю работу на себя.
*   **Смешанная (наиболее частая):** Клиент → рекурсивный запрос к локальному серверу. Локальный сервер → итеративные запросы к иерархии.

DNS‑сервер кэширует ответы на время (TTL), чтобы ускорить ответы на повторные запросы.

## 13.10. Протокол DHCP

**DHCP (Dynamic Host Configuration Protocol)** автоматизирует конфигурирование сетевых интерфейсов (IP‑адрес, маска, шлюз, DNS‑серверы).

*   **Ручной режим:** администратор жёстко привязывает MAC‑адрес к IP.
*   **Автоматическое статическое:** сервер выдаёт адрес из пула навсегда, запоминая соответствие.
*   **Динамическое распределение:** адрес выдаётся на ограниченный **срок аренды (lease time)**.

Упрощённая схема обмена:
1.  Клиент → **DHCPDISCOVER** (широковещательно).
2.  Сервер(а) → **DHCPOFFER** (предложения с IP).
3.  Клиент → **DHCPREQUEST** (выбор одного предложения).
4.  Сервер → **DHCPACK** (подтверждение аренды).

# Глава 14. Протокол межсетевого взаимодействия IP

## 14.1. IP‑пакет и функции протокола IP

IP — протокол без соединений, доставляющий пакеты «по возможности». Он не исправляет ошибки, а просто отбрасывает повреждённые пакеты.

**Формат заголовка IP‑пакета (IPv4)**

[IMG: 14.1 — Структура заголовка IP-пакета]
<img width="704" height="348" alt="image" src="https://github.com/user-attachments/assets/b902fb92-2936-479b-98ea-d9b82930daef" />

*Как найти: «ipv4 header format fields», «заголовок IPv4»; должна быть детальная схема всех 32-битных слов заголовка, размеры полей в битах.*

Основные поля:
- **Version (4 бита):** версия (4).
- **IHL (4 бита):** длина заголовка в 32‑битных словах (обычно 5, т.е. 20 байт).
- **Type of Service (8 бит)/DS‑байт:** для указания требований к QoS (приоритет, минимизация задержки/пропускной способности/надежности).
- **Total Length (16 бит):** общая длина пакета (макс. 65535 байт).
- **Identification (16 бит):** уникальный ID для сборки фрагментов.
- **Flags (3 бита):** бит **DF** (не фрагментировать), бит **MF** (есть ещё фрагменты).
- **Fragment Offset (13 бит):** смещение фрагмента в исходном пакете (кратно 8 байтам).
- **TTL (8 бит):** время жизни (макс. число хопов).
- **Protocol (8 бит):** код протокола верхнего уровня (TCP=6, UDP=17, ICMP=1).
- **Header Checksum (16 бит):** контрольная сумма заголовка.
- **Source/Destination Address (по 32 бита):** адреса отправителя и получателя.
- **Options (переменной длины):** необязательное поле для отладки (запись маршрута, строгая/свободная маршрутизация от источника).
- **Padding:** выравнивание заголовка до 32 бит.

### Алгоритм просмотра таблицы маршрутизации (без масок)

При поступлении пакета с адресом `IP_D`:
1.  **Поиск специфического маршрута к хосту:** ищем полное совпадение `IP_D` с записью в таблице. Если нашли — берём Next Hop.
2.  **Поиск маршрута к сети:** из `IP_D` выделяем номер сети и ищем его в таблице. Если нашли — Next Hop.
3.  **Маршрут по умолчанию:** если есть, используем его. Если нет — пакет отбрасывается.

**Важно:** Маршрутизатор никогда не собирает фрагменты в более крупный пакет.

## 14.3. Маршрутизация с использованием масок

Маски позволяют делить одну сеть на несколько подсетей (subnetting) и задавать сети произвольного размера.

### Алгоритм просмотра таблицы с масками

1.  **Специфический маршрут:** совпадение `IP_D` с записью, у которой маска `255.255.255.255` (/32).
2.  **Неспецифический маршрут:** для каждой записи вычисляется `IP_D AND Маска`. Результат сравнивается с адресом сети в записи. Совпавшие строки помечаются.
3.  **Выбор маршрута:**
    *   Нет помеченных строк и нет default — пакет отбрасывается.
    *   Одна помеченная строка — используется она.
    *   **Несколько помеченных строк (перекрытие адресных пространств)**: выбирается маршрут с **наибольшей длиной маски** (наиболее специфический).

## 14.5. Фрагментация IP‑пакетов

Фрагментация нужна, когда пакет переходит из сети с большим **MTU (Maximum Transmission Unit)** в сеть с меньшим MTU. Её выполняет **маршрутизатор**. Сборку фрагментов выполняет **только конечный узел**.

[IMG: 14.13 — Фрагментация в составной сети]
<img width="834" height="689" alt="image" src="https://github.com/user-attachments/assets/e86dc3ae-db38-4d20-beea-55275f567ea6" />


*Как найти: «ip fragmentation example diagram», «фрагментация ip пакета схема»; показан хост в сети MTU=4080, отправляющий большой пакет через маршрутизатор в сеть Ethernet MTU=1492. Маршрутизатор делит пакет на 4 фрагмента.*

Все фрагменты имеют одинаковый **идентификатор пакета**. Сборка ведётся по полю **смещения фрагмента** и флагу **MF**. Признак последнего фрагмента — `MF=0`. Узел-получатель при получении первого фрагмента запускает таймер сборки. При истечении таймера все фрагменты отбрасываются.

## 14.6. Протокол ICMP

**ICMP (Internet Control Message Protocol)** — вспомогательный протокол для оповещения отправителя об ошибках и диагностики сети. ICMP-сообщения инкапсулируются в IP-пакеты.

[IMG: 14.14 — Формат ICMP-сообщения]
<img width="615" height="320" alt="image" src="https://github.com/user-attachments/assets/50d2c1bd-3712-43b3-be47-c83a62fc9865" />

*Как найти: «icmp message format», «формат ICMP сообщения»; 8-байтовый заголовок: Type (1B), Code (1B), Checksum (2B), остальное зависит от типа.*

Две группы сообщений:
1.  **Сообщения об ошибках** (тип 3 — «Узел недоступен», тип 11 — «Истечение времени» и др.).
2.  **Сообщения «запрос-ответ»** (тип 8 — «Эхо-запрос», тип 0 — «Эхо-ответ», тип 13/14 — «Запрос/Ответ отметки времени»).

**Пример использования ошибок: утилита traceroute.** Отправляет серию IP-пакетов с монотонно возрастающим TTL. Когда TTL становится 0 на очередном маршрутизаторе, тот отбрасывает пакет и шлёт ICMP-сообщение «Истечение времени». Traceroute запоминает адрес маршрутизатора-источника этого ICMP-сообщения. Так выявляется весь маршрут.

**Эхо-протокол и утилита ping.** Утилита ping отправляет ICMP Echo Request (тип 8). Узел-получатель обязан прислать Echo Reply (тип 0). Кроме проверки достижимости, замеряется время оборота (RTT). В запросе и ответе присутствуют идентификатор и порядковый номер, позволяющие клиенту сопоставить ответы с запросами.
